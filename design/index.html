Expanded design

Deployment Studio as the EdgeCommons deployment control plane

This document treats deployment as a first-class EdgeCommons capability. The point is not to invent a fourth platform, or to hide platform reality behind a lowest-common-denominator abstraction. The point is to let a user describe a running solution in one semantic model, validate that model against the runtime contracts EdgeCommons already ships, and then produce the exact platform-native artifacts needed by host/supervisord, Kubernetes, and AWS IoT Greengrass.

The design therefore has two centers of gravity. The first is a stable domain model: solution, environment, edge node, component instance, config layer, secret reference, policy, renderer, release, and evidence. The second is a product workflow: author intent, preview render output, evaluate risk, generate or apply through controlled integrations, and observe whether the edge fleet converged.

1. Thesis: deployment is the missing layer above runtime configuration

EdgeCommons already makes component code portable across runtime services. A component can ask for messaging, configuration, state, health, metrics, hierarchy, credentials, and command handling without embedding every platform detail into the business logic. The deployment problem is the adjacent problem: somebody still has to decide which components run together, which identity each one uses, how config is layered, which storage and network resources are required, how secrets are referenced, and how all of that becomes deployable artifacts.

Today that knowledge is scattered across templates, recipes, Helm-like chart fragments, Docker Compose and supervisord examples, component READMEs, test infrastructure, and human memory. The scattering is tolerable while examples are small. It becomes expensive when a production edge solution has many component instances, multiple lines or cells in the same site, local broker topology, durable stream buffers, Greengrass policy constraints, Kubernetes probes, per-target storage, and a safety process that expects reviewable change history.

Deployment Studio should introduce a typed deployment definition that compiles to platform artifacts. The compiler metaphor matters. The source definition remains concise and semantic. The renderer is deterministic. Validation is layered. The output is inspectable. The release carries evidence. When a user changes the source, the system can show both the semantic diff and the rendered impact before any edge device is touched.

Semantic source

Store intent: components, topology, config layers, policies, secrets, targets, and promotion rules.

Native output

Emit artifacts operators already understand: INI, YAML, recipes, deployment JSON, lock files, and scripts.

Deterministic evidence

Tie every release to source commit, renderer version, artifact hashes, validation results, and apply history.

2. Current state: what the deployment layer must preserve

The design is grounded in the current implementation rather than a hypothetical greenfield platform. The core library already has a platform resolver, a strict runtime config schema, config-source selection, messaging and health conventions, component templates, and platform-specific test infrastructure. Those are the contracts the deployment layer should compile into.

The critical runtime distinction is that HOST, KUBERNETES, and GREENGRASS are EdgeCommons runtime profiles, while supervisord, Helm, Kustomize, GDK, Greengrass deployments, and CI/CD systems are deployment mechanisms. For example, a supervisord artifact should still launch a component with --platform HOST. A Greengrass recipe should still launch with --platform GREENGRASS and use GG_CONFIG. A Kubernetes workload should keep the ConfigMap whole-volume mount and the health probe conventions already used by the test chart.

3. Domain model: the source file is not a pile of target YAML

The source model should describe a solution at the level where users make deployment decisions. A user thinks in sites, lines, gateways, components, routes, certificates, device-local directories, broker topology, namespaces, device identities, approvals, and blast radius. Target YAML is a consequence of those decisions, not the authoring surface.

A deployment definition can be represented as a folder so that large solutions remain reviewable. The root document names the solution and environments. Component instance documents hold runtime and deployment intent. Config layer documents hold ordered hierarchy scopes and component leaves. Target profile documents capture platform-specific constraints that are legitimate platform facts rather than arbitrary overrides.

Illustrative source shape

This is intentionally smaller than the rendered output. It names the desired running system, pins component artifacts, references config lineage, and declares target policies.

The model must also carry negative space: what does not belong in the deployment source. Secret values do not belong there. Fully rendered Kubernetes manifests do not belong there unless they are release snapshots. Arbitrary shell scripts do not belong there unless they are declared as controlled hooks with platform, phase, inputs, and review policy. This keeps the definition understandable and makes the UI credible: the UI edits model fields, not blobs.

4. Configuration and ConfigComponent: authored layers, strict runtime output

Configuration is the place where deployment abstraction can either become useful or become dangerous. EdgeCommons runtime clients already expect a single effective config document that validates against the canonical schema. The deployment layer can make authoring more humane, but it must not loosen that contract. The safe rule is simple: users author deployment intent and raw config layers; renderers and clients produce one effective runtime config; the effective config is what gets validated and observed.

Hierarchical config has shipped in all four languages, so this chapter describes an existing contract rather than a proposal. ConfigComponent serves a raw lineage bundle: an ordered array of layers, each carrying a scope claim and a config fragment, ending in exactly one component layer. The SDK client merges layers[].config in array order — later layers win, objects merge recursively, arrays and scalars replace wholesale, an explicit null replaces — and validates only the merged effective document against the canonical schema. Layer metadata (id, kind, scope) never reaches that document. The server never pre-merges and never validates a partial layer as if it were a component config.

Two properties of the shipped contract shape the authoring surface directly. First, the hierarchy is user-defined: hierarchy.levels is any depth the customer wants — enterprise, site, building, zone, line is only the common example — and it must end with device. Second, the device is not a catalog node. It resolves from the running platform's thing name, so the catalog's nodes stop above it and components are the leaves. The client also enforces ownership: a child layer may not overwrite an ancestor's scope value or an ancestor-owned identity value, and an invalid reload keeps the previous effective config rather than notifying listeners with a broken one. Deployment Studio authors into those rules and should make lineage visible without forcing users to read the final JSON first.

ConfigComponent is the natural bridge from static files to centralized configuration. Its catalog already loads through a source abstraction with file, ConfigMap, and env providers. A deployment system can extend that seam with a Git-backed catalog source, or it can render Git content into the existing file/ConfigMap sources. The first path gives direct version awareness and branch/ref pinning. The second path is simpler and keeps the server unchanged, but shifts Git loading into CI/CD or the deployment renderer.

Authoring view

Users edit scope layers and component leaves with schema-aware forms, ownership badges, examples, and lineage-aware diffs.

Runtime view

Components consume one effective config through FILE, CONFIGMAP, GG_CONFIG, or CONFIG_COMPONENT sources.

Audit view

Release evidence records the Studio's hash of each authored layer file, the effective config hash, the validation result, and the version and provenance the bundle itself carries.

Important boundary

A component that bootstraps from ConfigComponent cannot depend recursively on ConfigComponent for its own bootstrap configuration. Deployment Studio should flag this while editing, before a user discovers the loop on an edge device.

5. Storage and versioning: Git is the audit log, not merely a backup

Deployment definitions need diff, review, history, rollback, provenance, and separation between proposed and approved state. Git is a natural storage layer because it already supplies those primitives and integrates with the DevSecOps systems that will evaluate changes. The design should treat Git as the durable source of truth and use the UI as a structured editor and reviewer for that source.

A production workflow should write every proposed change to a branch. The UI can show the friendly semantic diff, but the underlying change is still a real branch and a real commit. Promotion should tag a commit or create a release object whose manifest points to the definition commit, the renderer version, artifact hashes, validation results, supply-chain attestations, and target apply records.

That release object correlates two independent streams; it does not fuse them. Config releases and artifact releases are versioned and reconciled separately, each with its own drift signal and its own rollback target, so a config change ships without reshipping the binary and a binary ships without rewriting config. The manifest records which version of each was in effect together — that is what makes a coalesced apply auditable — but it never forces the two to move as one. Fusing them would reproduce precisely the coupling the platform's own decision register rejects.

Recommended repository layout

Use intent folders for definitions and layer inputs; use generated snapshots only for approved releases or for environments that require GitOps pull controllers. Avoid committing every transient preview render because that creates noise and makes review harder.

What the UI controls

The UI creates branches, edits typed definition files, runs render previews, opens pull requests, and reads validation evidence. It should never make an unreviewed direct write to the protected release branch in production mode.

6. Rendering: target adapters preserve platform reality

The renderer layer is where the abstraction pays rent. It must hide repetitive mechanics without hiding important target-specific decisions. HOST/supervisord, Kubernetes, and Greengrass have different ownership models, health semantics, configuration delivery mechanisms, credential stories, and rollback behavior. A renderer should produce native artifacts and a normalized deployment plan, but it should not pretend those targets behave the same.

HOST / supervisord

Render process groups, priorities, restart behavior, wait gates, directory creation, local broker configuration, and EdgeCommons CLI commands. Treat supervisord as a HOST deployment strategy, not as a separate runtime platform.

Kubernetes

Render ConfigMaps, Deployments or StatefulSets, Services, probes, optional RBAC, optional ServiceMonitor, PVCs, and labels/annotations that connect the workload back to the deployment release.

Greengrass

Render recipes, GDK configuration, component artifacts, accessControl, lifecycle, runWith, configurationUpdate merge/reset, and cloud or local deployment commands.

Every renderer should also emit a machine-readable plan. The plan is the common currency for validation, UI display, policy checks, and deployment. It should describe creates, updates, deletes, restarts, config mutations, credentials touched, target identities, and rollback affordances. This plan gives the UI a stable model even when the generated files differ radically.

7. Target deep dives: the abstraction must respect each deployment world

The three supported targets are not cosmetic variants. They have different controllers, failure modes, security boundaries, and operator muscle memory. The deployment abstraction works only if it lets common EdgeCommons concepts stay common while giving each target enough native expression to be production credible.

HOST / supervisord

HOST deployment is process-oriented. The running unit is a gateway or container that starts a local broker and a set of component processes. The renderer should emit supervisord program groups with stable priorities, start and stop behavior, restart policy, logging, and wait gates. It should generate the exact EdgeCommons command line for each component, including --platform HOST, --transport MQTT, FILE config paths, messaging files, and thing identity. It also needs a host bundle contract: directories to create, files to stage, checksums to verify, supervisor validation commands, and health checks to run after reload.

The open design question for HOST is apply semantics. EdgeCommons can generate a portable bundle, but production organizations may already use Ansible, SSH, MDM, golden images, or a site operations process. Deployment Studio should support generate-only first, then a narrow apply adapter that stages files, validates supervisor config, reloads ordered programs, and records evidence without becoming a general-purpose remote execution platform.

Delivery evidence on HOST is settled by accepting what the target is. HOST fleets are development and small-deployment territory, and HOST has no control plane — so there is no automated source of "the bundle arrived" evidence, and the Studio does not invent one. Convergence views degrade honestly for control-plane-less targets: the delivery verdict is unverified, stated as such, never inferred and never rendered as healthy. A small inspection utility that examines a HOST deployment in place — staged files, checksums, supervisor state — and emits delivery evidence for developers or small sites is a plausible later component, built on demand, not a requirement of the model.

Kubernetes

Kubernetes deployment is API-object-oriented. The renderer should preserve the current EdgeCommons Kubernetes contract: ConfigMap source, whole-directory mounts, Downward API identity, health endpoints, Prometheus metrics, optional RBAC, optional ServiceMonitor, and StatefulSet/PVC when durable state is required. The model should not let a user casually produce a subPath ConfigMap mount that breaks hot-reload behavior, nor should it grant RBAC by default when the workload does not need API access.

Kubernetes also has a mature deployment ecosystem. Some environments will use direct kubectl or Helm from CI. Others will use Argo CD or Flux and expect Git to be the reconciliation boundary. Deployment Studio should require the environment to choose one ownership mode. Direct apply and GitOps handoff should not run silently at the same time, because that makes drift ambiguous and can cause controller fights.

Greengrass

Greengrass deployment is nucleus-managed. The renderer must emit recipes with concrete component versions, artifacts, lifecycle, dependencies, component configuration, accessControl, and runWith settings. It must also produce deployment JSON with target ARNs and configurationUpdate merge/reset operations. Greengrass security cannot be reduced to recipe syntax; the design has to account for IAM, TokenExchangeService, IPC authorization, artifact buckets, and the difference between local lab deployment and cloud deployment.

Those target ARNs are thing ARNs, never thing group ARNs. Every member of a thing group shares one deployment document, and IIoT edge devices each carry a unique configuration, so a group cannot express per-device intent. A definition's nodes therefore map one-to-one onto Greengrass deployments: N nodes means N deployments, one apply record per node in the plan, and failure that is partial per node rather than a single group verdict. This costs more API calls and buys back exactly the staged, selectable rollout the product surface already assumes — and it removes union semantics, where a device inheriting two groups would receive the merge of both.

Greengrass is also the target where release locking matters most. A cloud deployment should consume concrete component versions and artifact URIs, not a moving source directory. Deployment Studio should block apply when versions are not locked, when wildcard IPC permissions are unexplained, or when lab regression evidence is required but missing.

8. Infrastructure as Code: Deployment Studio should be an IaC author and consumer

Infrastructure as Code belongs in the center of this design because real deployments at scale are not just component config. A complete solution may need Kubernetes namespaces, service accounts, storage classes, network policies, image registry permissions, AWS IoT things, Greengrass components, S3 artifact buckets, IAM policies, IoT policies, secrets, KMS keys, observability resources, and CI/CD identities. Some of those resources are best generated directly by EdgeCommons renderers. Others should be delegated to established IaC tools such as Terraform, AWS CDK, Pulumi, Helm, Kustomize, or GitOps controllers.

The key design choice is to separate deployment intent from infrastructure ownership. Deployment Studio should be able to render IaC modules or inputs, import external IaC outputs, and show their relationship to the edge solution. It should not assume it owns an organization's VPC, cluster, registry, or identity foundation unless the user explicitly chooses that mode.

How IaC marries with the UI

The UI should present infrastructure dependencies as first-class cards on the release plan. A Kubernetes target might show that a namespace and PVC class already exist, a ServiceAccount will be generated, and an ExternalSecret is expected from another system. A Greengrass target might show that each node's thing is provisioned, the artifact bucket is managed by Terraform, and the recipe is generated by the current release. A HOST target might show that OS package installation is outside the platform while supervisord config and local directories are generated by EdgeCommons.

In generate-only mode, the UI creates a pull request containing definition changes, rendered artifacts, IaC module inputs, and a plan summary. In apply mode, the UI still writes to Git first, but it may also trigger an approved runner that performs Terraform plan/apply, CDK synth/deploy, Argo CD sync, Greengrass deployment creation, or host orchestration. The UI should display the external system's plan and status, not replace that system's audit log.

9. DevSecOps integration: generation is useful, controlled deployment is different

Deployment Studio can begin as an artifact generator, but the architecture should leave room for controlled deployment execution. The difference is governance. A generator needs deterministic rendering, validation, and review. A deployer needs identity, authorization, audit trails, secrets discipline, policy enforcement, environment protection, concurrency control, rollback strategy, and live drift detection.

The safest architecture is Git-first. Even when the platform eventually effects deployments, the UI should produce a branch, a semantic diff, a rendered plan, and a validation bundle. The actual apply can be performed by a signed runner, GitHub Actions environment, Terraform Cloud/HCP run, Argo CD sync, Greengrass API call, or host orchestrator. Deployment Studio then becomes the connective tissue: it knows EdgeCommons semantics, it prepares native artifacts, it records evidence, and it correlates live status back to the definition.

Policy and evidence model

Rendered plans should be policy inputs. OPA/Rego-style policy is a good fit because the plan can be structured JSON: target platform, resources, permissions, images, secret refs, command lines, network exposure, Greengrass accessControl, Kubernetes securityContext, and host process users. Policies can produce blocking decisions, warnings, required approvers, and remediation hints.

Supply-chain evidence should travel with releases. For container and component artifacts this includes image digests, SBOM references, vulnerability scan results, signatures, provenance attestations, and the exact builder identity. For generated deployment artifacts this includes the source commit, renderer version, renderer inputs, output hashes, policy decisions, and apply target identity. Release evidence is what lets an operator answer: what did we intend, what did we generate, who approved it, what applied it, and what actually ran?

10. Product surface: detailed UI flows, screens, and interaction model

The UI cannot be a prettier YAML editor, and it cannot be a small set of cards that merely restate requirements. It has to be a working mental model for a difficult operational job: understand what is running, propose a desired-state change, keep hierarchical config explainable, preview target-specific artifacts, pass governance gates, and then watch the release converge.

The product should feel like a deployment control room. It needs a stable site tree, a release/draft context, a task workspace, and an inspector that explains why the system thinks something is safe or risky. The operator should always know four things: where they are in the fleet, which desired-state draft they are editing, which platform targets will be affected, and what evidence is still missing.

Primary user journeys

The UI should be optimized around jobs a platform or site operator actually repeats. These journeys are intentionally narrower than "edit deployment YAML" because each one has a different cognitive burden, a different risk profile, and a different correct default. The product can still store everything in Git, but the UI should lead with intent, consequences, and evidence.

Create a deployment draft

Entry is the current release lock for a site, line, or set of nodes. The UI creates a Git branch, freezes the base release, and shows dirty fields separately from rendered artifact changes.

Start with an empty system

First run is a guided setup path: create or connect storage, define hierarchy levels and first instances, set the enterprise target standard, assign and configure components for selected edge nodes, and produce a lab render before any live deployment is possible.

Add or replace a component

The user selects a component from the registry, chooses runtime capability needs, maps instances, then sees standard-target compatibility and exception impact before any artifact is generated.

Change hierarchical config

The user edits the lowest appropriate scope, sees the winning layer and overridden ancestors, and cannot accidentally hide an enterprise or site policy behind a device-local patch.

Review render consequences

The render review groups selected-target impact as restart, storage, network, identity, permission, config, and artifact changes. Raw generated files are always available, but they are not the first explanation.

Investigate drift

Drift is classified as definition, artifact, applied target, or runtime evidence drift. That distinction keeps a GitOps sync delay from looking like a broken component.

Promote or apply

Promotion writes a release lock and evidence bundle. Apply, when enabled, is a separate protected lane with credentials, approvals, audit append, target dispatch, and rollback visibility.

Hierarchical config authoring flow

The hierarchy editor is the most important mock because it is where the model can either become clear or become another hidden configuration system. It should be built around value explanation. A user starts with an effective value, expands to see the lineage that produced it, chooses the correct layer to edit, and only then changes the raw fragment. This matches the platform direction in the hierarchical config design: ConfigComponent resolves ordered ancestors, clients merge one effective config snapshot, and validation happens against the effective runtime document.

  1. Search by runtime path, common label, component field, or artifact consequence. For example, searching "heartbeat" finds `/heartbeat/intervalSecs`, the current winner, and the selected target artifacts affected by changing it.
  2. Show the lineage as enterprise, site, building, zone, line, device, and component. The UI should not assume this exact list is universal; it renders whatever ordered parent chain the deployment definition resolves.
  3. Mark each value as inherited, overridden, blocked, or locally owned. A blocked value explains the policy rule and the layer that attempted the illegal change.
  4. Let the user edit only the layers they own. Enterprise policy, site identity, and device binding can be visible without being editable in a production draft.
  5. Re-render effective config and target artifacts after every saved edit, then show validation and drift as evidence attached to the draft.

11. Operations: drift, rollback, and live convergence

Once Deployment Studio can generate a release, it must also help operators understand whether that release converged. Edge Console already points in this direction with health, fleet state, command gateways, and configuration review seams. Deployment Studio should connect those signals to the release model. A component is not merely "running"; it is running a specific artifact digest, a specific config hash, under a specific release tag, on a specific target platform.

Drift detection differs by target because delivery evidence is a per-target capability, not a given. In Kubernetes, GitOps controllers and the Kubernetes API compare desired and live objects. In Greengrass, the cloud deployment status per thing is the source of truth. HOST/supervisord has no control plane and therefore no automated delivery evidence: its delivery verdict is unverified, shown as exactly that — a distinct state, not "none" and not a pass. Each drift signal is computed only where the target class supplies evidence, and every verdict names its evidence source. An optional HOST inspection utility that emits delivery evidence is demand-gated, not part of the model.

Rollback

Rollback means selecting a previous release manifest and using the target renderer or external deployer to reapply its artifacts, not guessing from current files.

Hotfix

Hotfixes still create a branch and evidence bundle. Emergency apply can bypass normal timing, but not audit, identity, or artifact hashing.

Drift

Drift is shown as definition drift, artifact drift, apply drift, and runtime drift so operators can distinguish intent, generated files, target state, and process health.

12. Runtime and hosting: where Deployment Studio itself runs

Every other chapter describes what the Studio produces. This one describes what the Studio is: a service, with a process, a host, and dependencies. Leaving that unstated is not a neutral omission — it is the silence that lets a cloud creep in through the back door, one managed service at a time. The project began as a Greengrass-specific tool, and the constraint now is explicit: cloud execution is acceptable, dependence on a particular cloud is not, and the whole system must run on a laptop for development without incurring cloud cost.

The lock-in lever is state, not compute

It is tempting to answer "where does it run" with a compute choice, but compute is the easy thing to move. What binds a control plane to a vendor is what it depends on: a managed database, a queue, a function runtime, a hosted identity service. A Studio that needs a managed Postgres is locked in no matter how portable its code is, and "run it locally" degrades into "run a database locally too".

Chapter 5 already made the decision that dissolves this problem; it simply did not draw the consequence. Git is the database. Definitions, layers, releases, evidence, and approvals all live in a repository. That becomes a hard rule here: the Studio's only durable state is the Git repo, and every other piece of state it holds is a derived cache in an embedded SQLite file that can be deleted and rebuilt from Git at any time. There is no third datastore. Two consequences follow, and both are features rather than compromises: a single replica is the correct deployment (the Git host already performs concurrency control, which is also the answer to concurrent drafts), and local development means pointing the binary at a folder.

Five ports, each with a zero-cost local adapter

The Studio is a hexagonal core with exactly five ports. The rule that keeps it honest, and that belongs in code review rather than in prose: no cloud SDK may be linked above the port boundary. The Greengrass adapter may depend on the AWS SDK; nothing else in the system may. This is what prevents the tool's Greengrass ancestry from quietly reasserting itself as an AWS product.

The Git port deserves a specific note, because it is the one that usually forces a cloud dependency in designs of this shape. Running Gitea in the local compose stack means the entire governance story the design leans on — pull requests, protected branches, CODEOWNERS over the layers/ tree, approval before merge — is exercised offline, on a laptop, with no GitHub account and no network. The governance model is therefore testable at zero cost, not merely asserted.

One artifact: a static binary that is also a container

The Studio ships as a single self-contained binary with the UI embedded in it, and as an OCI image wrapping that binary. A container is the only deployment unit with no cloud-specific API surface: the same image runs on ECS, Cloud Run, Azure Container Apps, Fly, a customer's on-prem Kubernetes, a Docker host, or a NUC on a plant floor. The rejected alternatives are rejected on exactly this ground — functions-as-a-service force per-cloud packaging and event models, managed workflow engines force a per-cloud orchestration dialect, and managed databases force a per-cloud data dependency. Alongside the image, the Studio should ship a Helm chart, a docker-compose file, and a supervisord conf, the last of which means it is deployed by its own HOST renderer.

One honest cost to record: no EdgeCommons component publishes a container image today. Where a Dockerfile exists at all it is built for CI and side-loaded into a test cluster, never released, and edge-console ships no image. The pattern for containerizing a Rust EdgeCommons component is proven, but the Studio would be the first to publish a released image. That is a real, small piece of new work — release plumbing, not an existing capability to consume.

The stack, and why it is not a new one

The Studio is an axum service in Rust that serves an embedded React + Carbon single-page app, with a shared TypeScript protocol package defining the wire contract between them. This is not a novel choice; it is the third instance of a shape the org has already proven twice. edge-console is exactly this — a Rust gateway crate, a React/Carbon ui, and a shared protocol package, built on the edgecommons Rust library. (Its original TypeScript server was removed; any document still describing edge-console as a Node service, including its own citations in this folder's review, is stale.) config-component is likewise Rust.

Choosing Rust here is what turns the design's most fragile assertion into a structural guarantee. Determinism is load-bearing for every hash, lock, and evidence claim in this document, and the surest way to lose it is to reimplement the config merge. The Studio does not reimplement it: it computes effective config by calling core/libs/rust's config/layered.rs — the same code path the runtime executes — which the shared hierarchical-config-test-vectors/ already bind to the other three languages. Likewise, config-component's CatalogSource seam and catalog hashing are Rust, so a GitCatalogSource lands in the same language as the code it extends. Policy is Rego, compiled to WASM and evaluated in-process (or via a Rust Rego engine), which keeps OPA as the policy language while requiring no OPA server, no sidecar, and no hosted policy service — and it works offline.

This does not trigger the four-language parity rule. The Studio is a tool, not the core library, and config-component being Rust-only is the precedent. Parity is triggered only if evidence fields migrate into the core wire contract, which chapter 11's control-plane-derived evidence stance is specifically designed to avoid.

The CLI is the product; the server is a shell around it

edgecommons deployment validate | render | plan | diff | release must perform the entire model-to-artifact job with no server and no network. The server adds only branch and draft orchestration, the UI, evidence correlation, and access control. This single rule pays for most of the constraints in this chapter at once: the first two implementation slices ship with no hosting decision paid for at all, local development is free by construction, air-gapped and intermittently-connected sites work without a special mode (a CLI and a Git bundle on removable media is a complete workflow), and CI is just the same binary invoked in a job.

This resolves the CLI's language, and the answer is to rewrite it in Rust. The reasoning is not aesthetic. The deployment kernel must be Rust for the reuse and determinism reasons above, so the binary exists either way; keeping the Python CLI and shelling out to that binary would therefore require users to install both a Python runtime and the binary, and would leave a trampoline between them to maintain. The rewrite is also unusually cheap and unusually low-risk right now: the CLI is roughly 1,300 lines of command code whose only dependencies are jsonschema and GitPython; it has no boto3 dependency, because its AWS and Greengrass work is already shelled out to gdk and greengrass-cli; and its templating is not Jinja but literal token substitution over files copied from core/templates/. Copy files, substitute tokens, validate JSON, shell out — each has a boring Rust equivalent. Decisively, it is not published: installation is pipx install . from a checkout, so there is no PyPI package and no installed user base to break. This will never be cheaper than it is now.

The payoff extends past this project. Today, scaffolding a Java or TypeScript component requires a Python runtime — a real adoption tax on the front door of a deliberately polyglot ecosystem. A single static binary removes it. Two properties must survive the port: templates are currently bundled into the package so scaffolding works offline, and the interactive/conditional prompt logic is covered by tests that should be ported as behavior rather than redesigned. The existing pytest suite is the oracle for the rewrite, and the acceptance gate is the org's standing one — scaffold, build, and run all four language templates across the testable deployment options — not "the Rust tests pass".

Live evidence, without inventing an aggregation tier

Hosting and evidence are one decision, not two, because the fleet and drift screens are exactly what a hosting choice makes possible or impossible. The resolution has two halves. First, the Studio's default mode is Git-only: desired state, renders, releases, and drift measured against Git. No live plane, no aggregation tier, fully offline; the fleet views degrade to desired state plus last-known evidence, and say so, rather than presenting evidence they cannot obtain. Second, live evidence — when an environment wants it — arrives through a snapshot port rather than a new service tier: each site's edge-console, which is already on the bus and already maintains a fleet model, exports a periodic signed site-state snapshot, which the Studio ingests via a file drop, an S3-compatible bucket, or MQTT. These are three adapters, not three architectures. This composes with edge-console's deliberately site-scoped design instead of contradicting it, requires no cross-site cloud service, and requires no change to the core wire contract.

The explicit no-list

Stated as constraints, because "it started as a Greengrass tool" is precisely how each of these re-enters a design without anyone deciding to admit it.

  • No Lambda, Step Functions, or EventBridge as the pipeline spine — plain CI jobs operating on the normalized plan JSON.
  • No DynamoDB, RDS, or managed cache — Git plus an embedded SQLite derived cache.
  • No provider-specific identity — OIDC only.
  • No "S3 the service" in the core — the S3-compatible API, behind the blob port.
  • No AWS types in the domain model. apiVersion: edgecommons.io/v1alpha1 must never grow a top-level thingArn; Greengrass concepts live in the Greengrass adapter alone.
  • The Studio must never require being inside the cluster it deploys to.
  • No vendor CD product as the reconciliation boundary. Argo CD and Flux are acceptable: CNCF, self-hostable, and already assumed by chapter 7.
  • No multi-tenant hosted SaaS, which would force the identity, audit, and tenancy work that is already the blocking gap elsewhere, and would contradict PN-2.

13. Implementation roadmap: extend the existing system in layers

The design lands in slices that keep the current runtime and tests stable. The sequencing follows from chapter 12: because the CLI is the product and the server is a shell around it, the first two slices require no hosting decision to be paid for — they have no service, no identity model, and no cloud footprint of any kind. The hosting decision is nonetheless made up front, in chapter 12, rather than deferred; deferring it is what would otherwise leave the fleet, drift, and observe screens as permanent mocks.

Slice 1 builds one renderer, not three. The HOST renderer comes first because it can be checked against an oracle that costs nothing to obtain: any existing multi-device HOST site that a human has already hand-maintained — its supervisord process groups, its per-component config, its ConfigComponent catalog — is a set of reviewed, working files. The acceptance test is regenerating those exact files from a definition and comparing them byte for byte. That exercises the whole model — multi-device site, hierarchy, config lineage, per-component config — on shipped substrate with no cloud control plane at all. HOST is also where the marginal value is highest, since Kubernetes and Greengrass already have native deployment mechanisms and HOST has none.

  1. Kernel and CLI: the deployment model, the Rust kernel crate, and the HOST renderer, emitting native artifacts plus the normalized plan. New code, no migration risk, no server. Acceptance: regenerate an existing hand-maintained site and diff against the files a human wrote.
  2. Evidence: artifact hashes, validation bundles, and release manifests tied to Git commits, produced in CI. Still no service.
  3. Greengrass renderer and CLI port: recipes, GDK config, deployment JSON, and the release-lock governance showcase against a real Greengrass device. Port the six existing CLI commands onto the Rust binary and retire the Python CLI, with the existing test suite as the oracle.
  4. Storage and Kubernetes: Git-backed definition repositories, the Kubernetes renderer, and either rendering catalogs into the existing file/ConfigMap sources from CI or a ConfigComponent GitCatalogSource that reads pinned refs safely.
  5. UI: the Studio service from chapter 12 — axum plus the embedded Carbon SPA — beginning as a read-only pair of screens (config layers and render review) over Git, with identity delegated to the Git host.
  6. Execution and convergence: connect CI/CD, Terraform/CDK/Pulumi, Argo CD or Flux, Greengrass APIs, and host orchestrators behind explicit protected modes, then close the loop with drift, rollback, and the edge-console snapshot feed.

Two things stay out of the first product cut entirely: the prescriptive dataflow and capability model, which depends on runtime surfaces that do not exist, and artifact import, which is reverse-compilation and is research-grade. A descriptive dataflow graph derived from existing configs is tractable and remains useful.

Decision surface

            
What is now settled, and what is still open.

          

          

            

              settled — ch. 12
              Where the Studio runs
              
A single static axum binary with an embedded Carbon UI, shipped as one OCI image, on any container host or a laptop. Git is the only durable state; SQLite is a rebuildable cache. Five ports, each with a zero-cost local adapter.

            

            

              settled — ch. 12
              Studio vs Edge Console hosting
              
Standalone for v1, rather than hosted inside Edge Console. They share the UI stack, the brand tokens, and the evidence formats — not a process. Two fleet trees exist temporarily, and that is acceptable while one is explicitly desired-state and the other observed-state.

            

            

              settled — ch. 12
              CLI language
              
Rewrite the CLI in Rust and make it the single binary. The kernel must be Rust regardless, so shelling out from Python would force users to install both. It is ~1,300 lines, has no boto3 dependency, and is not published — so there is no user base to break.

            

            

              settled — ch. 12
              Live evidence
              
Git-only is the default mode; no aggregation tier is built. Where live evidence is wanted, each site's Edge Console exports a signed site-state snapshot through the snapshot port. Control-plane-derived first — no core wire change.

            

            

              Generated artifacts in Git
              
Commit render snapshots only at release boundaries, or keep them ephemeral and store hashes plus evidence. Leaning: commit at release boundaries, since GitOps handoff requires it anyway.

            

            

              Secret handling
              
Store references and policies, never secret values. Render ExternalSecret, mounted files, or Greengrass config refs.

            

            

              Override boundaries
              
Allow platform-specific overrides only where the semantic model cannot express a platform requirement.

            

            

              Promotion gates
              
Require evidence bundles for schema, semantic, renderer, and target smoke checks before a definition can be tagged. Open: whose infrastructure runs "target smoke", since not every environment has a spare device or cluster to smoke against — the gate must degrade rather than block.

            

            

              ReleaseLock shape — decided: two-stream
              
Config releases and artifact releases are independently versioned and independently reconciled, each with its own drift signal and rollback target. The ReleaseLock is a correlation and evidence envelope over both, not an atomic apply unit: it records what was in effect together without forcing the two to move together. A config change therefore ships without reshipping the artifact, and the reverse. Fusing them was rejected as the Greengrass coupling RM-002 explicitly refuses.

            

            

              Greengrass target granularity — decided: per-thing only
              
Thing groups are not used. IIoT edge devices each carry a unique configuration, and the members of a thing group necessarily share one deployment document, so a group cannot express per-device config. A definition's nodes map one-to-one onto Greengrass deployments, the union-semantics problem disappears, and per-node partial failure is native — which is what the staged rollout wanted anyway.

            

            

              Stream separation — decided: a model invariant, not a transport one
              
Greengrass does not technically fuse config and binary: a deployment can change config against an unchanged component version, or the binary while retaining config. It is the tooling and UI that combine them. So the model, release streams, evidence, drift, and rollback keep the two strictly distinct, while the Greengrass adapter may coalesce them into a single deployment where that suits the command; the ReleaseLock's correlation envelope keeps a coalesced apply auditable against both streams.

            

            

              HOST delivery evidence — decided: degrade honestly, no receipt machinery
              
Delivery evidence is a per-target capability. Kubernetes and Greengrass have control planes that report it; HOST does not, and HOST is development and small-deployment territory — so the Studio does not mandate receipts or agents there. Convergence views show HOST delivery as unverified, stated as such, never inferred. An inspection utility that examines a HOST deployment and emits delivery evidence is a possible later component, built on demand.

            

            

              Config delivery — decided: per provider, ConfigComponent preferred but not required
              
Customers may keep the native Greengrass config source or any supported provider, so the deployment system depends on no single component. The config stream is one model with N delivery adapters: the effective config is computed once by the shared layered-config code, and the component's declared config source selects how it reaches the device — a catalog push, a config-only deployment, a ConfigMap, a staged checksummed file, unit environment, or a shadow update. This is also why restart impact is a first-class field of the plan: a pure config update does not reliably restart a component — the reason EdgeCommons has dynamic config — and whether hot reload applies is a property of the config source, not the platform. File and ConfigMap changes are watched; an env change is not.
